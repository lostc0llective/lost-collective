## Success criteria

Brett asked for four things. This plan ties each to a verifiable outcome.

| Brett asked for | How we prove it |
|---|---|
| Understand when admin defaults conflict with custom CSS | `CONFLICTS.md` exists, typed A-E, each conflict has a file:line reference |
| Assess which rule should win | Each conflict in CONFLICTS.md has a "Verdict" column naming the winning rule and the reason |
| Eliminate conflicting rules or render them correctly | Phase 1-4 below delete or rewrite every rule with a P0/P1 verdict |
| Build like a high-end boutique agency | Phase 5 visual QA + Phase 6 production readiness checklist |

## Phasing philosophy

Six phases, each shippable to staging independently. Each phase has an entry gate (preconditions) and an exit gate (done criterion). If an exit gate fails, the phase rolls back on its own branch without blocking the next.

```
Phase 0: Safety net (branches, tags, staging push, rollback tested)
Phase 1: Dead-code deletion (zero behaviour change expected)
Phase 2: Hardcoded hex -> var references (visual parity expected)
Phase 3: !important scope-tightening (bucket C-1/C-2/C-3)
Phase 4: Admin-CSS wiring restoration (A-series fixes, visible admin changes)
Phase 5: Visual QA sweep (10 template types, desktop + mobile)
Phase 6: Production readiness (staging approved, push live, monitor)
```

## Phase 2 — Hardcoded hex -> var references

**Entry gate:** Phase 1 complete and on staging.

**Work:**

Target: every hardcoded hex in `custom.css` listed in CONFLICTS.md Type D.

1. **D1 `#4d4d4d`** — 9 occurrences. For each, classify:
   - Is it a body-text use? Replace with `var(--element-text-color--body)`.
   - Is it a surface/background? Introduce `--color-surface-slate: #4d4d4d` in `:root` and reference.
   - Is it the mobile header bg? That's the E1 case — handle in Phase 4.
2. **D2 `#ebac20`** — 4 occurrences + 15 admin settings use the same value. Plan:
   - Introduce `--color-brand-gold` in `:root`. (Rename from `--color-brand-yellow` for clarity — yellow is ambiguous.)
   - Update `replace_all` across custom.css to use `var(--color-brand-gold)`.
   - Defer the admin-side alignment to Phase 4 (A1/A6 fix).
3. **D3 `#6f6f6f`** — 4 occurrences. Replace with `var(--element-text-color--body, #6f6f6f)`.
4. **D4 `#f5f5f5`** — 3 occurrences. Decide role:
   - If it's a surface background, introduce `--color-surface-light`.
   - If it's a footer text colour, reference the relevant Flex footer var.
5. **B6 token collapse** — `--color-mid` and `--color-body-text` both hold `#6f6f6f`. Delete `--color-mid`. `replace_all` references.

**Exit gate:** `grep -c "#ebac20\|#4d4d4d\|#6f6f6f\|#f5f5f5" assets/custom.css` returns a number at least 80% lower than the pre-Phase-2 count. Staging renders pixel-identical. Pick any Flex setting that should now feed through (e.g. `regular_color` in admin, change to `#ff00ff`, verify body text goes magenta on staging, revert).

## Phase 3 — `!important` scope-tightening

**Entry gate:** Phase 2 complete and on staging.

**Work:**

Target: CONFLICTS.md Type C buckets C-1 (~150) and C-2 (~95). Leave C-4 (~38 genuine) alone.

1. Group custom.css `!important` declarations by (selector, property).
2. For each group:
   - **C-1 case** (fighting Flex `{% style %}` scoped to `#section-id`): rewrite the custom.css selector to include `#section-id` prefix, drop `!important`. Verify specificity math (`#section-id .foo` = 110 vs `.foo` = 10; the former wins without !important).
   - **C-2 case** (fighting another `!important`): reconcile by specificity; usually the lower-specificity one can drop `!important` and add a parent-scope qualifier.
3. C-4 keeps `!important` — but add a `/* WHY: inline style=... */` comment above each.

**Exit gate:** `!important` count in `custom.css` drops from 378 to ~80-120 (C-4 bucket + any that survive reconciliation). Staging renders pixel-identical. Spot-check the Header section (92 !important -> target <15).

## Phase 4 — Admin-CSS wiring restoration

**Entry gate:** Phase 3 complete and on staging.

**Work:**

The A-series fixes from CONFLICTS.md. These are intentionally visible admin changes — merchant setting now controls what they think they control.

1. **A1 / B7 fix** — delete `assets/custom.css:119` (`--color-primary: var(--color-brand-yellow)`). Change admin `button_primary_bg_color` to `#ebac20` (gold, matches current rendered appearance). Verify primary buttons render identically to before, but now driven by admin.
2. **A5 / E1 fix** — expose `mobile_header_background` in `settings_schema.json`. Wire through `snippets/head.styles.tokens.liquid` (or create a new snippet) to emit `--color-header-bg-mobile: {{ settings.mobile_header_background }}`. Update custom.css mobile header selectors at L247, L563, L589 to use `var(--color-header-bg-mobile)`. Set default to `#4d4d4d` in schema so current behaviour preserves.
3. **A6 fix** — decide the hover colour story. Recommended: admin `link_hover_color` stays gold; custom.css stops hardcoding `#ebac20` on its own hover selectors and references the Flex-owned var. Requires mapping which Flex CSS var comes from `link_hover_color` — check `snippets/head.styles.legacy-settings-color.liquid`.
4. **A2 fix** — body text colour. Replace custom.css's `--color-body-text` and `--color-mid` with direct usage of `var(--element-text-color--body)`. Delete those two token definitions.
5. **A3 fix** — heading colour. Rename `--color-dark` to `--color-accent-ink` *only if* it's used in non-heading contexts. For heading selectors specifically, switch to `var(--element-text-color--heading)`.
6. **A4 fix** — footer background. Two options per CONFLICTS.md A4 verdict. Pick one, document in the refactor commit.

**Exit gate:** merchant changes every top-level colour setting in admin (primary button, body text, heading, link hover, mobile header, footer bg) one at a time. Each change propagates to the rendered page on staging within one reload. Revert all to current values.

## Risk register

| Risk | Mitigation |
|---|---|
| A Phase 1 dead-selector deletion turns out to be live in a rare template | Re-verification step in Phase 1 requires rendering 4 template types. If a selector is live on a template we didn't check, rollback tag + staging revert is 1 command. |
| A `!important` removal in Phase 3 breaks a Flex section we didn't map | Each Phase 3 batch commits separately. Rollback is git revert of that commit, `shopify theme push`. |
| An admin-CSS re-wire in Phase 4 reads a Flex var name that changed between v5.2.1 and v5.5.1 | Phase 0 established the new var names by reading `snippets/head.styles.*.liquid`. Cross-reference every var name in CONFLICTS.md against those files before starting Phase 4. |
| Merchant enters admin during the refactor and their changes don't propagate | Staging and production are separate theme slots (Phase 0 decision). Merchant changes apply only to the slot they're editing. |
| The refactor introduces a regression that only shows on a template we forgot to check | Phase 5's 20-capture sweep is the firebreak. If something slips through, post-push monitoring in Phase 6 catches it within 24h. |
