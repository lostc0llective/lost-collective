# Theme Settings <-> CSS Conflict Audit

> Generated by Cowork 2026-04-18 against `feat/flex-migration` HEAD (Flex v5.5.1 + LC customisations).
> Raw data: `audit/*.txt`, `audit/*.tsv`, `audit/*.json`.
> Framework reference: `audit/WALKTHROUGH.md` (conflict types A-E explained).

## The core problem in one paragraph

Flex v5.5.1 has a proper design system. Admin settings (`config/settings_data.json`) feed through Liquid snippets (`snippets/head.styles.*.liquid`) and `layout/theme.liquid` into CSS custom properties like `--color-primary`, `--element-text-color--body`, `--element-button-radius`. Flex's own CSS consumes those vars. Merchant changes a value in the theme editor -> it propagates. `assets/custom.css` runs a **parallel, hardcoded, disconnected** token system in its own `:root` block (lines 56-120), re-declaring the same colours with literal hex values that do not re-read admin. The same hex values exist in both places today, so the site looks correct, but any theme-editor change silently fails to propagate into anywhere `custom.css` touches. That is the source of "I changed it in admin and nothing happened" and "I fix one thing and another breaks". Everything below is a specific instance of that pattern.

## Severity scoring

- **P0** — currently misleading Brett (admin setting appears to work but CSS overrides it) OR breaking accessibility.
- **P1** — duplicate values that are in sync today but will drift the first time admin is touched.
- **P2** — dead code or stylistic inconsistency, no current user impact.

## Type C — `!important` inflation (378 total -- 92 just in the Header section)

`!important` is a cascade override. Each one should defend against something specific. 378 across 4,360 lines means 8.7% of declarations are screaming. That's unsustainable.

**Breakdown by section (from the `:root`-adjacent analysis of custom.css):**

| Section | !important count | Likely cause |
|---|---:|---|
| 6. Header | 92 | Fighting Flex's `{% style %}` injected styles inside `snippets/header-*.liquid` which have `#section-id` specificity |
| 17. Footer | 47 | Same pattern as header -- Flex injects scoped section styles |
| 42. Accessibility (2026-04-09 audit) | 47 | Forced hit-target sizes, focus outlines; harder to justify without seeing selector specificity |
| 5. Navigation | 42 | Mega menu and dropdown battles |
| 18. Collection Page | 31 | Flex collection-grid component overrides |
| 34. Homepage Category Grid | 26 | Custom grid layout fighting Flex section defaults |
| 41. Mobile Homepage | 22 | Media-query overrides against Flex mobile defaults |
| (others) | 71 | Lower density |

### C-bucket verdicts

| Bucket | Rough share | Action |
|---|---:|---|
| C-1: `!important` defends against Flex's `{% style %}` block that uses `#section-id` scope. Keep-but-scope-tighter is the right move -- restructure the custom.css selector to include `#section-id` too, dropping `!important`. | ~40% (~150 rules) | Safe removal in Phase 3 of REFACTOR-PLAN. |
| C-2: `!important` defends against another `!important`. Pure war. Usually both can be removed if the specificity is reconciled. | ~25% (~95) | Audit-as-you-go in Phase 3. |
| C-3: `!important` on a single-declaration utility-ish rule with no competition -- gratuitous. | ~25% (~95) | Safe to delete in Phase 1. |
| C-4: `!important` genuinely needed because the declaration being fought IS inline `style="..."` or a user-agent default. | ~10% (~38) | Keep, comment why. |

Specific examples to kill early (low-risk `!important` deletions):

- `custom.css:150` -- `font-family: Montserrat, 'Montserrat Fallback', sans-serif !important;` on `.navbar-link, .header__link, h1, h2, h3, h4`. Admin already sets these fonts via `heading__font` -> `--element-text-font-family--heading`. Replace with `font-family: var(--element-text-font-family--heading);` and drop `!important`. If that breaks, the selector list is the problem, not the importance.
- `custom.css:87-92` -- `--radius-xs: 3px;` etc. `!important` on custom-property definitions does almost nothing (Custom properties' cascade doesn't behave like other props in the way most devs expect). Audit for any `!important` on a `--var-name: value` declaration and delete. ~~None found in custom.css from a grep; noted in case refactor introduces them.~~

## The quick-win list (do these first)

If Brett only takes seven actions from this audit:

1. **Fix A1 / B7** -- delete `custom.css:119` (`--color-primary: var(--color-brand-yellow)`). Decide what colour primary buttons should be and set `button_primary_bg_color` in admin accordingly. This single line currently overrides Flex's entire button system.
2. **Fix A5 / E1** -- expose `mobile_header_background` as a schema setting, wire it to `--color-header-bg`. Stop lying about `header_background` being the global setting.
3. **Fix B5** -- delete `--color-sale` (unused). Done.
4. **Fix B1-B4** -- delete `--color-white`, `--color-facebook`, `--color-twitter`, `--color-pinterest` from custom.css. Flex already provides them.
5. **Fix D3 / A2** -- replace every `#6f6f6f` literal and every `var(--color-mid)` / `var(--color-body-text)` usage with `var(--element-text-color--body)`. One global find/replace in custom.css.
6. **Fix D2** -- standardise on one name for the gold colour. Rename `--color-brand-yellow` to `--color-primary` (or whatever you land on at A1), then delete the custom definition and let Flex's settings flow through.
7. **Fix C-3** -- delete gratuitous `!important` on type declarations (L150, property-definition lines if any). Estimated ~95 deletions with low risk.

## What you walk away with
