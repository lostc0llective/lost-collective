# Dead Code Audit

> Generated by Cowork 2026-04-18 against `feat/flex-migration` HEAD.
> Companion to `audit/CONFLICTS.md` and `audit/REFACTOR-PLAN.md`.
> Scope: declarations, selectors, settings, and files that exist but do nothing. Pure cruft.

## What counts as dead

Three tests:

1. **Declared but never referenced** — a CSS custom property defined in `:root` that no `var(...)` lookup consumes; a setting in `settings_schema.json` that no Liquid or CSS touches; a snippet that no `{% render %}` includes.
2. **Targets something that no longer exists** — a CSS selector for a class or ID that nothing in the rendered HTML carries.
3. **Defends nothing** — an `!important` declaration on a property that has no competing rule at any specificity.

All three types are safe to delete in Phase 1 of `REFACTOR-PLAN.md`. The risk is zero *if* the detection is correct. The sections below label each item with its detection method so Brett's CC sprint can verify before deleting.

## 3. Orphaned `!important` declarations (C-3 bucket)

The "defends nothing" category from CONFLICTS.md Type C. Estimated ~95 rules of ~378 total.

Detection recipe:

```bash
# For each !important in custom.css, check the declaration's property against
# the Flex source. If nothing else in the codebase sets that property on a matching
# selector, the !important is defending nothing.
```

Examples to delete in Phase 1:

| File:Line | Declaration | Why dead |
|---|---|---|
| `custom.css:150` | `font-family: ... !important;` on `.navbar-link, .header__link, h1-h4` | Admin setting writes to `--element-text-font-family--heading`; no competing `!important` found. Replace with a var reference, drop `!important`. |
| Any `--custom-property: value !important;` on a custom-property definition line | The `!important` flag on a custom-property declaration has almost no practical effect because the var is read at usage site via the cascade, not inherited like a normal property | Delete `!important` everywhere it appears on a `--var-name: value` line. |

**CC action:** write a script that reads custom.css, groups `!important` declarations by (selector, property), then for each group checks whether any other rule in custom.css, the Flex stylesheet, or an inline `{% style %}` block targets the same selector+property. Groups with zero competition go on the `DEAD-IMPORTANT.txt` list for Phase 1 deletion.

## 4. Dead admin settings (in schema, never rendered)

Source: `audit/settings-schema-flat.json` (285 settings) cross-referenced with custom.css + layout/theme.liquid + snippets/head.styles.*.liquid for the setting's `id`.

The CONFLICTS.md A-series (A1-A6) is a related-but-different bug: those settings DO reach CSS but get overridden. **True dead settings** are ones whose `id` appears in no Liquid template at all — the merchant can set a value in admin and it gets written to `settings_data.json`, but nothing ever reads it.

Detection recipe:

```bash
# For each setting id in settings-schema-flat.json, grep the theme for usage
while read id; do
  count=$(grep -r "settings\.$id\|'settings.$id'" --include="*.liquid" . | wc -l)
  [ "$count" = "0" ] && echo "DEAD: $id"
done < <(jq -r '.[].id' audit/settings-schema-flat.json)
```

Known suspects to run through that filter:

- Legacy block entries for removed apps (`shopify://apps/instafeed/...` block in `settings_data.json:548`). Instafeed is uninstalled but the block reference remains as a dead entry.
- Any Klaviyo settings whose Liquid references were removed during the Phase 1-5 cleanup.
- Any settings with `type: "header"` or `type: "paragraph"` that were group labels for now-deleted settings.

**CC action:** execute the recipe above. Expected output is a list of 5-20 truly dead settings. Each one should be deleted from `settings_schema.json` with a matching entry in `settings_data.json` cleaned up.

```bash
# For each .liquid in sections/, check if any template (templates/*.json) references it
find sections/ -name "*.liquid" | while read f; do
  name=$(basename "$f" .liquid)
  count=$(grep -rl "\"type\":\s*\"$name\"" templates/ | wc -l)
  [ "$count" = "0" ] && echo "ORPHANED SECTION: $name"
done
```
